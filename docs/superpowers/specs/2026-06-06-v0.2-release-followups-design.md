# v0.2 Release Follow-Ups

## Goal

Record the known follow-up topics that remain after the `0.2.0` MCP surface
update so they are tracked as design direction rather than treated as release
blockers.

## Current Release Position

The `0.2.0` release is intended to ship the current command-discovery-oriented
MCP surface:

- tools for search, lookup, scanning, rendering, runtime inspection, and sample
  metadata
- command resources under `resource://commands/...`
- prompts for script inspection and command discovery workflows

The current branch is considered releasable with that surface.

## Follow-Up Topics

### 1. Public surface stability

The current resource and prompt surface is acceptable for `0.2.0`, but it is
still early enough that future refinement is plausible.

Known follow-up questions:

- whether the command resource URI structure should remain unchanged long-term
- whether a dedicated reverse-lookup tool should be added between
  `search_manual` and `lookup_command`
- how far reviewed command coverage should expand before a later stability claim

These are follow-up design and roadmap questions, not current release blockers.

Detailed subtopics:

- `resources`
  - whether `resource://commands/{command}` and
    `resource://commands/{parent}/{command}` should be treated as long-term
    stable URI forms
  - whether individual command resources need any additional summary or
    navigation fields beyond the current reviewed lookup payload
- `prompts`
  - whether the current two prompts are enough as the stable workflow layer
  - whether future prompts should remain guidance-only or start carrying more
    opinionated workflow structure
- `discovery flow`
  - whether a dedicated reverse-lookup tool should be added
  - how that future tool should relate to `search_manual` and `lookup_command`
- `coverage`
  - which command families should be expanded next before making a stronger
    stability claim around command discovery

### 2. Render device and output control

The current render surface is useful, but important device/output choices are
still hidden inside runtime defaults.

Known follow-up questions:

- whether render tools should expose Topdrawer PostScript orientation directly
- whether a server default orientation should be chosen explicitly
- whether Ghostscript controls such as DPI or output padding belong in the MCP
  argument surface
- how script-authored `SET DEVICE` lines should interact with server-managed
  output capture

This topic now has its own design note:

- `docs/superpowers/specs/2026-06-06-render-device-control-design.md`

This is a design and interface topic, not only an implementation detail. It is
also not yet a release blocker for `0.2.0`, because the current render flow is
functional even though it is not yet as controllable as desired.

Detailed subtopics:

- `td` wrapper policy
  - the runtime currently prepends a fixed `SET DEVICE POSTSCR FILE='...'`
    line
  - future work needs a clearer policy for which device options are fixed by
    the server and which can be supplied by the caller
- `orientation`
  - investigated examples suggest orientation is driven by Topdrawer device
    behavior before Ghostscript PNG conversion
  - the current default should be revisited explicitly rather than left as
    hidden implicit behavior
- `Ghostscript output policy`
  - DPI
  - padding around the cropped image
  - any future background or output-device changes
- `precedence and safety`
  - whether render arguments override script content
  - whether script-side `SET DEVICE` should be rejected, constrained, or
    merged under narrow rules

### 3. Render prerequisites and operator guidance

The runtime render path depends on external `td` and `gs` executables.

Known follow-up work:

- improve user/operator guidance for diagnosing render setup problems
- make `get_server_runtime_info` the explicit first troubleshooting step
- keep repository scope clear that this package does not install or wrap `td`
  itself

This is primarily a documentation and operator-experience improvement area.

Detailed subtopics:

- `troubleshooting flow`
  - make `get_server_runtime_info` the explicit first troubleshooting step
  - document the expected inspection order for `td` and `gs` resolution
- `setup guidance`
  - improve README guidance around what must already be installed
  - keep the repository boundary clear that this package does not install
    Topdrawer itself
- `failure interpretation`
  - clarify whether common failures are expected to come from:
    - missing executables
    - `td` runtime errors
    - Ghostscript conversion failures

### 4. Discovery-flow completion

The current `0.2.0` agent flow is coherent, but it still has one intentionally
missing middle layer.

Known follow-up work:

- decide whether to add a narrow reverse-lookup tool
- define what repository-owned data that tool should search
- keep the reverse-lookup path smaller than full manual search and narrower
  than a general search engine

This topic is connected to public-surface stability, but is worth tracking
separately because it directly affects how complete the command-discovery story
 feels in practice.

Detailed subtopics:

- `query source`
  - user wording
  - script wording
  - sample metadata
- `result shape`
  - candidate commands only
  - candidate commands plus short reasons
- `relationship to existing surfaces`
  - should remain narrower than `search_manual`
  - should feed into `lookup_command`

### 5. Coverage depth and promotion order

The current reviewed command coverage is enough for `0.2.0`, but not broad
enough to treat the command-discovery surface as mature across the wider manual.

Known follow-up work:

- continue expansion in staged, practical command-family slices
- preserve the current principle that coverage should expand where it most
  improves agent usefulness first
- avoid turning coverage growth into a broad parser or database project

Detailed subtopics:

- command families most relevant to existing script inspection
- device/data basics that matter for render and file reasoning
- lower-priority layout/styling areas that can come later

This topic overlaps with the broader coverage roadmap and should stay aligned
with it rather than becoming a disconnected backlog.

## Current Promotion Guidance

The current follow-up set is now split between:

- concrete tasks that are ready to promote into GitHub issues
- broader design topics that should remain in Superpowers docs for now

### Ready To Promote Into GitHub Issues

The following topics are concrete enough to track as standalone tasks:

- fresh-session workflow validation execution using the existing scenario assets
- render prerequisites and troubleshooting guidance improvements around `td`,
  `gs`, and `get_server_runtime_info`
- the next narrow reviewed-command coverage slice, when the target command
  family is chosen explicitly

These are suitable for issues because they can be implemented or verified
independently without first resolving larger roadmap questions.

### Keep In Superpowers Docs For Now

The following topics should remain in design/roadmap space until they become
more concrete:

- long-term public-surface stability questions for resources and prompts
- reverse-lookup design details beyond a narrow MVP scope
- future render-format expansion order such as `eps` versus `pdf`
- longer-term artifact-subsystem enhancements such as richer provenance or
  broader retention/configuration changes
- broader command-coverage and preprocessing roadmap questions

These are not yet good issue candidates because they still depend on design
choices, prioritization decisions, or scope narrowing.

## Non-Goals

- Do not reopen the `0.2.0` command resource redesign itself in this memo.
- Do not treat these follow-ups as a reason to keep the branch unreleased.
- Do not expand the repository scope into automatic external dependency
  installation.
- Do not treat this memo as a fully ordered delivery plan.
- Do not duplicate the full content of the more focused render-device design
  note here.
